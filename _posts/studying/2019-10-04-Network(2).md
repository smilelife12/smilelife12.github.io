## 네트워크
- - -
브런치의 정주홍님의 개념 정리[https://brunch.co.kr/@toughrogrammer/16](https://brunch.co.kr/@toughrogrammer/16)에 굉장히 정리가 잘 되어 있어서 보고 공부하였다.


### 전송 계층 정리
- - -
 소켓으로 데이터를 잘 전달할 방법에 대해서 다뤘다. 비용이 더 들지만 보낸 데이터를 그대로 받을 수 있는 TCP 프로토콜을 이용하거나, 낮은 비용이지만 신뢰성이 떨어지는 UDP 프로토콜을 선택할 수도 있다. 프로토콜을 통해 프로세스에 데이터를 잘 전달해 주는 것은 OS가 도와준다. 받은 데이터의 활용은 응용 계층에 달려 있다.


### 네트워크 계층(Network Layer)
- - -
 네트워크 계층의 역할은 전송 계층에서 보내는 패킷을 목적지 노드 까지 도착시키는 것이다. 중간 라우터를 통한 라우팅(Routing)과 포워딩(Fowarding), 필요하다면 연결 설정을 담당한다.  
 라우팅과 포워딩의 차이를 보면, 라우터는 다른 라우터들과 물리적으로 연결되어 있는 것 이다. 라우팅은 현재 라우터에서 연결된 라우터 중 어떤 라우터로 보내는 것이 최적인지 결정하는 것이다. 라우팅 알고리즘이 수행되고 나면 어떤 헤더를 가진 패킷을 어떤 링크로 전송하라는 정보가 정리된 포워딩 테이블이 만들어진다. 만들어진 포워딩 테이블을 기반으로 라우터는 전달받은 패킷을 다른 라우터로 포워딩한다.  
 ATM, Frame Relax, X.25와 같은 모델들은 다른 라우터까지의 대역폭을 예약할 수 있다. 데이터 그램이 전송 되기 전 두 호스트 간의 경로를 가상연결(Virtual Connection)로 설정하여 두 호스트가 충분한 서비스를 제공받을 수 있도록 한다.  
  네트워크의 프로토콜은 크게 라우팅, 인터넷, ICMP 프로토콜로 나뉜다. 라우팅 프로토콜은 포워딩 테이블을 만들기 위한 프로토콜, 인터넷 프로토콜은 주소 체계를 위한 프로토콜, ICMP 프로토콜은 에러 메세지를 전달받기 위해 주로 쓰는 프로토콜이다.


### 데이터그램 네트워크(Datagram Network)
- - -
 데이터그램 네트워크에서는 그때그때 받은 데이터그램의 목적지 주소를 확인하여 포워딩 테이블에 따라 출력링크로 전송한다. 포워딩 테이블의 각 항목은 주소 범위와 출력 링크를 가진다. 주소 범위인 이유는 IPv4의 주소 체계는 2의 32승, 40억개 가량을 가질 수 있기 때문이다. 범위가 아니라 각 주소로 만들면 테이블 크기가 너무 커진다. 받은 데이터그램에 대해 포워딩 테이블에 있는 엔트리 중 어떤 것을 선택할지 결정하기 위해 Longest Prefix Matching 방법이 이용된다.  


### 인터넷 프로토콜(Internet Protocol)
- - -
 인터넷 프로토콜은 호스트 주소 지정과 패킷 분할 및 조립 기능을 담당한다. 데이터 링크 게층에서 전달할 수 있는 데이터의 최대 크기를 MTU(Maximum Transmission Unit)이라 하는데, 데이터 그램이 이보다 크면 분할하여 offset으로 지정한다. 분할된 데이터그램들은 목적지에서 재조립된다. 인터넷 프로토콜부터는 IP주소가 나타난다. `192.168.0.1` 같은 주소는 IPv4 주소 체계에 의한 주소이다. IPv4는 전 세계적으로 사용된 첫 번째 인터넷 프로토콜이다. 현재는 고갈되어 IPv6으로 대체되고 있다.  
 네트워크망 상에서 호스트 또는 라우터 간의 물리 링크 연결을 인터페이스(Interface)라고 한다. IP주소는 호스트가 아닌 인터페이스에 부여된다. 라우터는 여러 노드들과 연결되어 있으므로, 여러개의 인터페이스를 갖고, 호스트는 보통 하나의 유선 혹은 무선연결이 되어 있으므로 하나 혹은 두개의 인터페이스를 갖는다. 실제 네트워크는 중간에 이더넷 스위치(Ethernet Switch)가 여러 개의 호스트와 연결 되어 있고 이더넷 스위치와 라우터가 연결되어 있다.  
 서브넷은 IP주소로 표현 가능한 주소가 너무많아 관리의 효율성을 위해 도입되었다. IP주소는 서브넷 부분(상위 비트)과 호스트 부분(하위 비트)로 나뉘게 된다.


### 서브넷(Subnet)
- - -
IP의 서브넷 부분이 같은 노드를 모은 것을 지칭한다. 중계하는 라우터없이 물리적으로 연결된 네트워크망이라고 할 수도 있다.  
서브넷은 관리상의 이점을 갖게 해준다. 하위 네트워크망인 서브넷의 관리를 맡기는 것을 서브네트워킹(Subnetworking)이라고 한다.  
서브넷을 활용하기 위해 IP 주소에서 어디까지가 서브넷 부분인지 알아야 한다. `223.1.1.0/24`와 같은 표현은 `223.1.1.0`주소의 상위 24bit가 서브넷 주소임을 알리며, `/24`를 서브넷 마스크라 한다. 24bit가 서브넷 마스크인 것을 `255.255.255.0`로 표현하며, 이를 이용해 IP주소와 Bitwise AND 연산을 하면 서브넷 부분을 얻을 수 있다.  
클래스 주소 체계는 IP주소의 서브넷을 8, 16, 24bit로 나누는 것이며, `/8`을 A클래스, `/16`을 B클래스, `/24`를 C클래스로 서브넷을 가진다. 이방법 대신 임의의 길이를 갖는 방법을 CIDR(Classes Inter Domain Routing)이라 한다.


### 호스트 주소와 주소 블록 획득
- - -
 호스트가 IP주소를 입력 받는 방법은 고정 IP와 유동 IP인 두 방법이 있다. 고정 IP는 호스트에 특정 IP를 지정해 버리는 방법이고, 유동 IP는 주소를 동적으로 획득하는 방법이다. 네트워크 연결 시만 주소를 가지므로, IP주소를 재활용 할 수 있다. 모바일 기기 같은 잦은 네트워크 접속 종료로 인해 매우 유용하다. 유동 IP를 지원하는 방법으로 DHCP(Dynamic Host Configuration Protocol) 프로토콜을 사용하는 것이다.  
 기관에서 여러 호스트를 관리하기 위해 `인터넷 서비스 공급자 ISP(Internet Service Provider)`에게 돈을 주고 산다. ISP는 더 상위의 IP 주소 관리단체 (Internet Corporation for Assigned Names and Numbers, ICANN)로부터 주소 블록을 발급받는다. 이 발급받은 주소를 쪼개 고객에게 주소 블록을 판다.  


### Hierarchical Addressing
- - -
 넓은 서브넷상으로 라우팅 된 패킷이 ISP 망 내에 도착하면 또 다시 더 좁은 서브넷으로 라우팅 되기를 반복한다. 이를 Hierarchical Addressing이라 한다.  
만약 다른 ISP로 이전하면 이를 이용하여 라우팅 할순 없다. 이 경우 ISP 인터넷 망에 추가적인 주소블록도 나타낸다. 더 좁은 서브넷은 서브넷의 길이가 더 길어질 것이므로 앞서 언급한 Longest Prefix Matching 방법에 의해 적절한 ISP로 라우팅 된다.


### NAT(Network Address Translation)
- - -
 NAT는 IP 주소 하나를 갖고 여러 호스트를 관리하는 방법이다. 지역 네트워크를 형성하고, 지역 네트워크 상 주소를 NAT 장비가 관리한다. 지역 네트워크에서 패킷이 외부로 나갈 때 NAT 장비를 거치고, 이 때 패킷의 출발 주소가 NAT 장비의 주소로 변경된다. 하나의 IP로 관리하기 때문에 출력하는 패킷의 포트 번호를 다른 번호로 바꾼다. 이러면, 나중에 지역 네트워크로 진입하려는 패킷의 목적지 포트 번호를 확인해 적절한 지역 네트워크 호스트를 찾게된다.  
 NAT는 내부의 IP 주소를 숨기기 때문에 보안상의 이점을 갖는다.


### ICMP(Internet Control Message Protocol)
- - -
 호스트와 라우터 사이의 네트워크 계층 정보를 통신하기 위해 사용되는 프로토콜이다. ping, traceroute 프로그램에서 쓰이는 것이 익숙하다.


### 라우팅 알고리즘
- - -
크게 Link State, Distance Vector, Hierarchical Routing 세 가지로 나뉜다. Link State는 주기적으로 링크의 주변 라우터들에게 알리면(Link state broadcast) 최소 비용 경로를 다시 계산하는 방식이다. 다익스트라 최단 경로 알고리즘이 쓰인다. 목적지까지의 경로를 알고 있으므로 라우팅 테이블 변화에 따른 전파속도가 빠른다. OTPF프로토콜이 대표적인 예이다.  
Distance Vector는 목적지 까지의 거리(Hop count)와 목적지까지 가기 위해 어떤 이웃 라우터를 거쳐야 하는지 방향(Vector)만 저장하는 방식이고, 벨먼-포드 알고리즘이 사용된다. 적은 정보만 저장하므로 메모리는 절약되지만, 라우팅 테이블에 변화가 생겼을 때 전파 시간이 오래걸리고, 정해진 시간마다 업데이트 하므로 트래픽 낭비가 크다. RIP 프로토콜이 대표적이다.
Hierarchical Routing은 Hirerarchical Addressing과 비슷하다. 인터넷을 AS(Autonomous System)라는 좀 더 작은 구역으로 나눈다. 각 AS는 관리자에 의해 운영되며, 각자의 라우팅 프로토콜을 사용한다. Intra-AS 라우팅 위한 프로토콜을 IGP(Interior Gateway Protocol)라고 하며, OSPF와 RIP가 주로 사용된다. Inter-AS 라우팅을 위해서는 먼저 다른 AS로 찾아갈 수 있도록 하는 라우팅이 필요하며, EGP(Exterior Gateway Protocol)이라 한다. BGP가 주로 쓰인다.  


### 네트워크 계층 정리
- - -
 전체 인터넷은 계층적 네트워크로 이루어지며, 하위 네트워크를 서브넷이라 한다. 라우터는 전달 받은 패킷의 목적지를 보고 포워딩 테이블에서 가장 적합한 인터페이스를 찾아 그 인터페이스로 패킷을 출력한다. 목적지 전부를 테이블로 만들지 않고, 서브넷들로 구성시켜 효율적으로 한다. 외부의 AS로 이동하기 위한 라우팅과 AS내에서 이동하기 위한 라우팅이 나뉜다. 공통적으로 라우터들은 서로 정보를 공유하며(Interworking) 최적의 라우팅을 위해 노력한다.


### 데이터 링크 계층과 물리 계층
- - -
데이터 링크 계층은 실제로 인터페이스를 통해 데이터 전송이 이루어진다. 데이터 링크 계층에서는 흐름 제어, 오류 제어, 반이중과 전이중 서비스를 제공한다. 인터페이스는 하나이고, 양측이 함께 사용하므로 형평성 있게 조율을 해야한다. 또한, 물리 계층에서 전달시 물리적 문제로 잘못된 비트가 전송될 수 있다. 이를 위해 오류 감지 서비스가 사용되며, 수정 기능이 있기도 하다.  
사용되는 주소는 MAC(Media Access Control) 주소이다. IP 주소는 바뀌지만, MAC 주소는 네트워크 인터페이스에 할당된 고유 식별자이다. IP 패킷을 전송하기 위해 물리적 네트워크 주소인 MAC 주소가 필요한데, 이를 모를 때 ARP(Address Resolution Protocol) 을 이용한다.  
물리 계층에서는 물리적 비트를 전송하기 위한 일을 한다. 하드웨어와 밀접하게 연관되어 있다. 이더넷, 토큰 링 등이 이 계층에 포함되며, 전기 신호 혹은 광 통신 등이 있다. 전기 신호가 이동하며 약해지는 것을 증폭시키기 위해 리피터가 사용된다. 모뎀이 이 계층에 속한다.  


### 세션 계층
- - -
 패킷이 올바르게 도착했다는 가정하에 시작된다. 네트워크 망을 통한 통신은 OS에 의해 관리되며, 실제로 우리가 프로그래밍하는 것은 OS 위에서 돌아가는 프로세스이다. 이 프로세스가 통신을 관리하기 위한 방법에 대한 계층이 세션 계층이다. 또한, TCP/IP 세션을 만들고 없애기도 한다.
 프로세스가 OS에게서 데이터를 전달받기 위해서는 소켓을 이용한다. 논리적 연결에 대해 다루는 계층이다.


### 표현 계층
- - -
 소켓으로 받은 데이터는 바이트들이다. 이러한 바이트 들을 데이터로 인식하기 위해 처리하는 계층이다. 번역기와 같은 역할을 하는 것이다.
 압축과 암호화 또한 여기서 사용되며, 압축을 통해 데이터를 작게 만들고, 보안을 위해 암호화 시키기도 한다.


### 응용 계층
- - -
 이해할 수 있는 데이터 들을 처리하는 위치이다. 이 응용 계층의 예는 전자 메일과 관련된 SMPT(Simple MailTransfer Protocol), 파일 전송을 위한 FTP(File Transfer Protocol), WWW 에서 주고받는 HTTP(HyperText Transfer Protocol)이 있다.  
 특히 HTTP는 가장 널리 쓰인다. HTML 문서를 받아오고, CSS 기반으로 페이지를 렌더링 하고, JS를 해석하여 실행함으로 웹을 제공한다. [참조 : Naver D2](https://d2.naver.com/helloworld/59361)를 참조하면 좋을 듯하다.  
 주의할 점은 프로토콜의 해석한 내용이 결과물이라는 것이다.RESTful API는 요청 결과를 JSON 포맷으로 리턴한다. 라이브러리를 이용할 떄 HTTP 응답의 Body를 JSON으로 파싱하여 Dictionary 객체를 얻기도한다. 실제로는 HTTP의 Body는 바이트들이므로 서버가 XML을 줫는데 JSON으로 파싱하려면 에러가 나게된다.
