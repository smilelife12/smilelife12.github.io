## 데이터베이스(DB)
- - -
브런치의 정주홍님의 개념 정리[https://brunch.co.kr/@toughrogrammer/17](https://brunch.co.kr/@toughrogrammer/17)에 굉장히 정리가 잘 되어 있어서 보고 공부하였다.


### 데이터베이스 사용 관점 개요
- - -
DB는 실제 데이터와 그 데이터에 대한 정보를 모아 놓은 메타 데이터로 이루어져 있다. 효율적인 관리와 접근 관리등 추가적인 기능을 위해 DB를 사용하게 된다.  
DB에서 원하는 데이터를 가져올 때 DBMS에게 질의(Query)하게 된다. 이는 SQL을 이용해 접근하며 작업을 수행한다.  
DB는 구조화된 데이터를 저장하며, 모델링을 필요로 한다. RDB 같은 경우는 관계형 데이터 모델링을 하며 보통 개체-관계 모델링(Entitiy-Relationship Modeling, ERD) 방법으로 데이터를 구조화한다. 이를 DB설계라고한다.  
중복 데이터가 존재하지 않도록 함수 종속성을 확인해 이상(Anomaly)이 존재하는지 알고, 이를 없애기 위해 정규화(Normalization)을 한다.  


### 데이터베이스 스키마
- - -
DB에서 데이터의 구조, 표현 방법, 데이터 간의 관계를 형식 언어로 정의한 구조이다. 널리 쓰이는 3단계 스키마 구조에선, DB를 관점에 따라 외부 스키마, 개념 스키마, 내부 스키마 3단계로 나뉜다.  
외부 스키마는 각 사용자의 관점에 대해 보는것, 개념 스키마는 모든 사용자의 관점으로 보는 것, 내부 스키마는 물리적으로 DB에 접근하는 관점으로 보는 것이다. 추상화와 비슷하므로 논리적/물리적 독립성을 얻게 된다.  


### 관계형 데이터 모델
- - -
DB를 관계의 집합으로 표현하였고, 관계를 행(Tuple)의 집합, 행을 속성(Attribute)의 집합으로 본다. RDB에서는 관계가 테이블(Table), 행이 레코드(Record), 속성이 컬럼(Column)으로 대치된다.  
관계형 데이터 모델에서 속성은 해당 속성이 가질 수 있는 모든 값에 대한 도메인을 갖고, 원자적이어야 한다. 원자적이란 말은 더이상 쪼개어질 수 없다는 말이다. 관계 간에는 순서가 없고, 행 간에도 순서가 없다(Unordered).  
RDB에서는 관계의 집합과 제약 조건의 집합으로 이루어졌다. 제약 조건으로 키 제약 조건(Key constraint), 개체 무결성 제약 조건(Entity integrity constraint), 참조 무결성 제약 조건(Referential integrity constraint) 등이 있다.
키는 슈퍼 키(Super key), 후보 키(Candidate key), 기본 키(Primary key)로 나뉜다. 슈퍼 키는 그 관계의 모든 행을 유일하게 식별 할 수 있도록 하는 속성의 집합이다. 가능한 적은 개수의 속성으로 키를 구성하는 것이 행을 식별하는 데 이점이 있다. 그래서 후보 키는 슈퍼 키중 가장 속성의 개수가 적은 것이다. 그 중에서도 하나의 후보 키가 기본키로 선택되고, 모든 관계는 단 하나의 기본키만을 가진다.  
참조 무결성 제약 조건은 RDB에서 매우 중요한 제약 조건 중 하나이다. 흔히 외래 키(Foreign key)에 적용된다. 외래 키의 대상 관계는 그 키를 가진 행이 반드시 존재해야 한다는 제약 조건이다. 이 무결성 제약 조검으로 관계있는 데이터가 반드시 DB에 존재한다는 보장을 할 수 있으므로 매우 유용하다.  


### 관계 대수
- - -
데이터 취급을 위한 연산 체계로 관계 대수(Relation algebra)가 사용된다. SQL 언어에서의 연산과 대치되며, DBMS가 질의 처리를 하기전 관계 대수로 인지한다.  
절차적 언어로 select, project, union, set difference, cartesian product, rename 6가지 연산으로 이루어져 있다. 거기에 대입, 교집합, 조인 등이 추가되어 있다.
- select(시그마) : 행을 골라내는 것으로 튜플을 선택하는 것
- project(파이) : 열을 골라내는 것으로 속성(어트리뷰트)를 골라내는 것
- union(합집합) : 합집합 연산
- set difference(차집합) : 차집합 연산
- cartesian product(x) : 두 테이블을 합치는 것
- join(리본 모양) : 공통 속성을 중심으로 두개의 릴레이션을 합치는 것
  - 세타 조인(theta join) : 조인 조건을 만족하는 튜플로 이루어진 릴레이션을 만듦
  - 동등 조인(equi join) : 세타 조인 중 동등한 조건을 갖는 조인
  - 자연 조인(natural join) : 한 기준을 가지고 동일한 속성 값을 모은 릴레이션 만듦
  - 외부 조인(outer join) : 하나의 속성을 기준으로 겹치는 것이 없으면 `null`로 표시하여 모든 릴레이션을 합쳐서 만듦


### SQL(Structured Query Language)
- - -
SQL은 프로그래밍 언어로 DBMS에 질의한다. 관계 대수 연산은 보통 `SELCECT ~ FROM ~ WHERE` 식의 질의를 한다.데이터 조작언어에 해당되며, 데이터 정의 언어, 데이터 제어언어로 SQL이 나뉜다.  
데이터 정의 언어는 테이블이나 색인, 제약 조건 등의 생성이나 삭제, 수정 작업에 대한 언어이다. RDB에서 관꼐는 테이블로 나타내며, RDBMS 테이블에서 데이터를 찾기 위해 색인을 만들고, 제약 조건들을 통해 데이터를 안정적으로 운영한다.  
데이터 조작 언어는 DB에 대해 검색, 추가, 삭제, 갱신을 위한 언어이다. SELECT, INSERT, DELETE, UPDATE로 시작하는 SQL이 데이터 조작언어에 포함된다.
- SELECT : 테이블에서 데이터를 추출하는 언어로 가장많이 사용된다. 여기에 추가적인 항목을 설정할수 있다.
  - `WHERE` : 어떤열을 불러올 지 지정
  - `GROUP BY` : 연산 함수가 각 그룹에 저장되도록 속성을 그룹열에 공유
  - `HAVING` : GROUP BY절에서 정의된 그룹들 중에서 검색
  - `ORDER BY` : 반환하는 열에 순서 지
```sql
SELECT [ALL | DISTINCT] 컬럼명 [,컬럼명...]
FROM 테이블명 [,테이블명...]
[WHERE 조건식]
[GROUP BY 컬럼명 [HAVING 조건식]]
[ORDER BY 컬럼명]
GROUP BY 컬럼명[,컬럼명...]
ORDER BY 컬럼명[,컬럼명...]
```
- INSERT : 테이블 또는 뷰에 한개이상의 행을 추가
```sql
INSERT INTO 복사 될 테이블명(컬럼명1, 컬럼명2,...) values(데이터1, 데이터2)
```
- UPDATE : 테이블이나 한 개 이상의 행을 바꾼다. 모든 행을 바꿀 때도 조건절을 사용해 가능하다.
```sql
UPDATE 테이블명 SET 컬럼명 = 데이터 [, 컬럼명 = 데이터 ...] [WHERE 조건]
```
- DELETE : 한개 이상의 행을 삭제, 조건을 적지않으면 모든행이 삭제된다.
```sql
DELETE FROM 테이블_또는_뷰_이름 [WHERE 조건]
```  


### VIEW
- - -
View는 테이블을 수정하지 않고, 데이터를 보는 관점을 정의하는 것으로 논리적 테이블이라고 생각할 수 있다. 테이블을 숨기고, 뷰를 통해 보이므로 보안을 추구할수 있다.  
실체회 뷰(Materialized View)는 자주 사용되는 뷰 데이터를 따로 저장해 질의 처리 속도를 향상할 수 있다. 즉, 실체화 뷰의 데이터는 물리적으로 존재한다. 캐시와 유사하다.  


### 무결성 제약 조건(Integrity Constraint)
- - -
DB에 입력되고 유지되는 것을 막을 수 있다. 제약 조건은 SQL, 데이터 정의 언어에 의해 추가, 삭제, 수정된다. 널리 사용되는 무결성 제약 조건으로 NOT NULL, UNIQUE, PRIMARY KEY, FOREIGN KEY, CHECK가 있다.NOT NULL과 UNIQUE는 어떤 값이 NULL이 아니어야 하고, 해당 관계 내에서 유일해야 한다. PRIMARY KEY는 앞서 언급한 키 제약조건과 같고, FOREIGN KEY는 외래 키로 참조 무결성 제약 조건을 만족시켜주기 위한 제약 조건이다. CHECK 제약 조건은 사용하면 어떤 데이터가 조건을 만족해야만 하도록 제한할 수 있다.  


### Trriger
- - -
DB관리의 편이를 위한 기능으로 Trigger의 선언과 삭제 또한 SQL로 이루어진다.  


### 권한 관리
- - -
DBMS가 여러 사용자에 의해 이용되는 시스템이기 때문에 사용자별 권한을 관리할 수 있도록 기능을 제공해주는 것이 필요하다. 리눅스에서 여러 사용자로 로그인하여 시스템을 이용하는 것과 비슷하다.  


### 개체-관계 모델(Entity-Relation Model)
- - -
DB를 잘 설계해야 관리가 쉽고 질의 수행 속도도 빨라진다.  
관계형 데이터 모델을 사용하는 곳에서 데이터 구조화를 위해 개체-관계 모델로 모델링 한다. 관계 등에 대한 설정을 해주며 일대일, 일대다, 다대다 관계가 있다. 보통 외래키를 두고 일대다, 다대다를 설정한다.  


### 정규화(Normalization)
- - -
정규화를 통해 크고, 제대로 되지 않은 테이블들을 작게나누어 데이터 중복을 없애고, 잘 조직된 구조로 만든다. 잘못된 테이블에서는 삽입 이상, 삭제 이상, 갱신 이상 등이 발생하며 연쇄적으로 발생할 수 있다.  
정규화는 단계적으로 제 1 정규형, 제 2 정규형, 제 3 정규형, 보이스-코드 정규형, 제 4 정규형, 제 5 정규형, 제 6 정규형이 있다.
정규화는 함수 종속성을 알아야한다.
> 함수 종석성은 릴레이션에서 두 개의 애트리뷰트 집합 간 제약의 일종이다. X와 Y를 부분 집합이라 할 때, X의 값 각각에 대해 시간에 관계없이 항상 애트리뷰트 Y의 값이 오직 하나만 연관되어 있을 때 Y는 X에 함수 종속이라 한다.  


### 제 1 정규형(1NF)
- - -
각 속성이 원자적이어야 한다. 기본 키로 각 행을 식별할 수 있어야 한다. 그리고 중복되는 항목이 없어야 한다.  



### 제 2 정규형(2NF)
- - -
 후보 키 전체로 후보 키에 속하지 않는 속성을 결정할 수 있어야 한다. 하나의 속성으로만 후보 키로 사용할 경우 제 2 정규형을 만족한다.


### 제 3 정규형(3NF)
- - -
제 2 정규형을 만족하면서 동시에 모든 속성이 기본 키에 대해서만 의존되어야 한다. 이 조건을 만족하면 추후 종속성이 발생 하지 않는다.  


### 보이스-코드 정규형과 제 4 정규형
- - -
보이스-코드 정규형은 강한 제 3 정규형이라고도 불린다. 모든 결정자가 후보 키여야 한다. 후보 키는 모든 행을 식별할 수 있는 최소의 속성 집합이다.  
제 4 정규형은 다치 종속성이 없는 것을 요구한다. 다치 종속은 어떠한 조건을 만족하는 튜플이 릴레이션 안에 있을 것을 요구한다. 다치 종속이 있을 경우, 데이터는 중복 되기 때문에, 이를 없애려는 것이다. 제 4 정규화 이상은 잘 하지 않으며, 너무 복잡해 진다.  


### 데이터베이스 사용 관점 정리
- - -
DB는 3단계 스키마로 계층적 분리를 통해 관리의 효율성을 도모한다. DB는 관리 시스템인 DBMS에 의해 관리된다. 사용자는 DBMS를 거쳐 DB를 사용하고, DBMS는 여러 사용자들이 함께 사용하는 시스템이다.  
관계형 데이터 모델은 데이터를 나타내기 위해 많이 사용된 모델이다. 이를 사용하는 DB를 관계형 데이터베이스라고 한다. 관계에 대한 연산은 관계 대수를 이용한다.  
SQL을 통해서 질의를 하며, 데이터 정의 언어, 데이터 조작 언어, 데이터 제어 언어로 나뉘며, 데이터 조작 언어는 관계 대수와 대치한다. DBMS는 뷰와 무결성 제약 조건, Trigger, 권한 관리등의 기능을 제공한다.  
모델링시 개체-관계 모델링 방법을 많이쓴다. 이를 통해 다이어 그램이 만들어지고 그를 이용해 RDB를 구현한다. 그리고 데이터를 관리하기 위해 정규화를 한다. 보통 제 3 정규형까지 진행한다.  


### 데이터베이서 개발 관점 개요
- - -
트랜잭션은 DB사용에 있어서 중요한 4가지 요구사항을 만족시켜주는 기능이고, DBMS는 여러 사용자에 의해 이용되므로 트랜잭션들을 동시에 잘 실행해야한다. 그리고 에러 발생시 복구를 잘 하는 것도 중요하다.  
데이터는 저장 장치에 기록되며, 저장 장치를 최대한 잘이용하여 데이터를 찾아야한다. 또한, 방대한 데이터 때문에 색인이 꼭 필요하며, DBMS에서 질의 처리를 위해 거치는 과정에 대해 알아보면서 실제 질의가 어떻게 실행되는지 알아본다.  


### 트랜잭션(Transaction)
- - -
상용 DB시스템들은 대부분 트랜잭션 기능을 제공한다. 트랜잭션은 하나의 논리적 작업이라고 볼 수 있다. DBMS는 여러 사용자에 의해 사용되므로, 여러 사용자들의 트랜잭션을 동시에 수행해야 성능 문제가 발생하지 않는다. 이 과정에서 실행중 장애가 발생할 수 있다.  



### 트랜잭션의 성질과 상태
- - -
트랜 잭션은 DB의 무결성을 지키기 위해 ACID 성질을 갖는다
- 원자성(Atomicity) : 트랜잭션 내의 모든 연산이 적용되거나, 아무것도 적용되지 않아야 한다.
- 일치성(Consisitency) : 트랜잭션 수행 전 DB가 무결한 상태였으면, 트랜잭션 수행 후에도 무결해야한다.
- 고립성(Isolation) : 여러 트랜잭션이 동시에 실행되고 있어도 사용자는 자신의 트랜잭션만 실행되고 있다고 느껴야 한다.
- 지속성(Durability) : 완료된 트랜잭션의 결과는 시스템 장애가 발생해도 DB에 반영되어야 한다.  
트랜잭션의 상태는 Active, Partially commited, Commited, Failed, Aborted 상태로 나뉜다.  


### 직렬 가능성(Serializability)
- - -
트랜잭션이 동시에 잘 실행하기 위한 것을 동시성 제어라고한다.  
트랜잭션을 올바르지 않게 실행할 경우 발생하는 문제는 Dirty read, Lost update, Unrepeatable read 세가지가 있다.
- Dirty read : 아직 완료되지 않은 값을 읽음
- Lost update : 트랜잭션에서 수정한 값을 다른 트랜잭션에서 수정해서 이전 갱신을 잃게 됨
- Unrepeatableread : 한 트랜잭션이 처음 읽은 값과 나중에 다시 읽었을 때의 값이 달라지는 문제로 어떤 값을 읽을 때마다 다른 값을 읽을 수도 있게 되는 문제이다.  
동시성 제어는 위 세가지 문제를 방지하는 방법이다.  
트랜잭션의 연산들을 순서대로 실행하는 것을 스케쥴이라 할때, 각 트랜잭션을 순서대로 실행한 것을 직렬 스케쥴(Serial schedule)이라고 한다. 앞서 말한 세가지 문제가 없는 스케쥴을 직렬 가능 스케쥴(Serializable schedule)이라고 한다.  


### 충동 직렬 가능성(Conflict serializability)
- - -
동일한 데이터에 대해 두 연산이 이루어 지면, 최소 하나의 연산이 쓰기 연산이면 두 연산은 충돌한다고 한다. 경쟁조건과 유사하다.  
비 충돌 연산은 서로 순서를 바꿔도 결과 차이가 없지만 충돌 연산은 실행 결과가 달라진다. 스케쥴에서 비 충돌 연산의 순서를 바꾸어 직렬 스케쥴이 되면 충돌 직렬 가능 스케쥴이라 한다.  


### 뷰 직렬 가능성(View serializability)
- - -
같은 트랜잭션들로 이루어진 스케쥴 S1, S2에 대해
- S1의 트랜잭션 Ti에서 처음 읽은 데이터 Q값과 S2의 Ti가 Q에 대해 같은 값을 읽고
- S1이 Ti가 쓰기 시작한 값을 다른 트랜잭션 Tj가 읽는다면, S2에서도 마찬가지로 Ti가 수정한 값을 Tj가 읽고
- S1의 마지막 쓰기 연산이 S2의 마지막 쓰기연산과 동일하다면  
이 스케쥴을 뷰 동치라 한다. 직렬 스케쥴과 뷰 동치인 스케쥴이라면 그 스케쥴을 뷰 직렬 가능 스케쥴이라고 한다.
